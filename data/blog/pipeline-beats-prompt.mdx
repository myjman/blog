## 프롬프트 잘 쓰면 장땡일까

"프롬프트 엔지니어링이 미래 역량이다."

2023년부터 이 말을 수없이 들었다. 맞는 말이다. 같은 모델이라도 프롬프트를 어떻게 쓰느냐에 따라 결과가 달라진다. 그래서 사람들이 프롬프트 기법을 공부하고, 템플릿을 모으고, 구조화된 프롬프트를 연구한다.

그런데 1년 넘게 AI로 코딩하면서 느낀 게 있다. **프롬프트 실력보다 파이프라인 설계 능력이 훨씬 중요하다.**

프롬프트를 아무리 잘 써도 매번 손으로 입력해야 한다면 한계가 있다. 반면 평범한 프롬프트라도 자동으로 돌아가면 잠자는 동안에도 일한다.

---

## 프롬프트의 한계

프롬프트 엔지니어링은 **한 번의 상호작용을 최적화**한다. 질문 하나에 답변 하나. 요청 하나에 결과 하나.

문제는 현실의 일이 한 번으로 끝나지 않는다는 것이다.

블로그 글을 쓴다고 해보자. 프롬프트로 초안을 뽑을 수 있다. 하지만 그 다음은? 교정하고, 포맷팅하고, 이미지 넣고, 파일 생성하고, 커밋하고, 배포해야 한다. 프롬프트 하나로 끝나는 게 아니다.

코드 리뷰도 마찬가지다. PR 하나를 AI에게 리뷰 요청할 수 있다. 그런데 매일 10개씩 올라오는 PR을 매번 복붙해서 요청하고 있나? 그건 프롬프트 엔지니어링이 아니라 노동이다.

프롬프트 실력이 상위 10%라고 해도, 그 프롬프트를 하루에 5번밖에 못 돌리면 평범한 프롬프트를 100번 돌리는 사람을 이길 수 없다.

---

## 파이프라인이란

파이프라인은 **작업이 자동으로 흘러가는 구조**다.

입력이 들어오면 처리되고, 처리 결과가 다음 단계로 넘어가고, 최종 결과물이 나온다. 사람이 중간에 개입하지 않아도 된다. 개입할 수는 있지만, 안 해도 돌아간다.

예를 들어 이런 파이프라인이 있다:

```
[매일 아침 9시]
→ 주제 파일에서 오늘 주제 확인
→ Claude Code로 블로그 글 작성
→ prettier로 포맷팅
→ git commit & push
→ 완료 체크
```

한 번 만들어 놓으면 매일 아침 자동으로 블로그 글이 생성된다. 프롬프트는 그 파이프라인의 한 부품일 뿐이다.

---

## 생산성 격차가 벌어지는 곳

프롬프트 실력의 차이는 **선형적**이다. 프롬프트를 2배 잘 쓰면 결과 품질이 조금 좋아진다. 10배 잘 쓰면 꽤 좋아진다.

파이프라인의 차이는 **지수적**이다. 파이프라인이 없으면 사람이 일한 만큼만 나온다. 파이프라인이 있으면 사람이 안 일해도 나온다.

| 방식   | 프롬프트 실력 | 하루 실행 횟수 | 결과물       |
| ------ | ------------- | -------------- | ------------ |
| 수동   | 상위 10%      | 5회            | 고품질 5개   |
| 수동   | 보통          | 10회           | 보통 10개    |
| 자동화 | 보통          | 100회          | 보통 100개   |
| 자동화 | 상위 10%      | 100회          | 고품질 100개 |

물론 마지막 줄이 제일 좋다. 하지만 세 번째 줄이 첫 번째 줄을 이긴다. 보통 실력으로 100개 만드는 사람이 고수가 5개 만드는 것보다 낫다.

특히 AI 시대에는 품질 격차가 줄어들고 있다. 모델이 좋아지면서 "보통 프롬프트"의 결과물도 쓸 만해졌다. 격차가 벌어지는 건 실행 횟수다.

---

## 파이프라인을 굴리는 사람들

이미 파이프라인을 굴리는 사람들이 있다.

**콘텐츠 제작자** — 매일 유튜브 스크립트를 AI로 생성하고, 썸네일 초안을 만들고, SNS 게시물을 발행하는 파이프라인. 하루에 5개 채널을 운영하면서 영상 편집만 직접 한다.

**개발자** — PR이 올라오면 자동으로 AI 리뷰가 달리고, 테스트가 돌고, 문서가 업데이트되는 파이프라인. 코드 품질 체크를 사람이 안 한다.

**리서처** — 키워드를 넣으면 관련 논문을 찾고, 요약하고, 정리된 노트가 노션에 쌓이는 파이프라인. 자료 조사가 자동이다.

**마케터** — 경쟁사 뉴스를 크롤링하고, 요약하고, 분석 리포트가 이메일로 오는 파이프라인. 모니터링을 AI가 한다.

이들의 공통점은 **프롬프트를 매번 직접 치지 않는다**는 것이다. 프롬프트는 파이프라인 안에 미리 들어가 있다.

---

## 파이프라인 구성 요소

파이프라인을 만들려면 몇 가지가 필요하다.

**트리거** — 언제 실행할지. 시간(매일 9시), 이벤트(PR 생성), 수동(버튼 클릭) 등.

**입력** — 무엇을 넣을지. 파일, 데이터베이스, API 응답, 사용자 입력 등.

**처리** — AI가 할 일. 여기에 프롬프트가 들어간다. 프롬프트는 파이프라인의 한 부품이다.

**출력** — 결과를 어디에 보낼지. 파일 저장, 커밋, 이메일, 슬랙, 노션 등.

**에러 처리** — 실패하면 어떻게 할지. 재시도, 알림, 로깅 등.

도구는 많다. n8n, Make, Zapier 같은 노코드 자동화 툴. GitHub Actions, cron 같은 개발자 도구. [Claude Code](/blog/vibe-coding-terminal)같은 터미널 에이전트를 스크립트로 호출할 수도 있다.

---

## 프롬프트 vs 파이프라인

|             | 프롬프트 엔지니어링           | 파이프라인 설계        |
| ----------- | ----------------------------- | ---------------------- |
| 최적화 대상 | 한 번의 상호작용              | 전체 워크플로우        |
| 실행 주체   | 사람                          | 시스템                 |
| 확장성      | 선형 (사람이 실행해야 늘어남) | 지수 (자동으로 늘어남) |
| 한계        | 사람의 시간                   | 비용, 도구 제약        |
| 핵심 역량   | 모델 이해, 언어 감각          | 시스템 설계, 자동화    |

둘 다 중요하다. 하지만 둘 중 하나만 골라야 한다면? 파이프라인이다.

프롬프트 실력은 시간이 지나면 평준화된다. 모델이 좋아지면 "대충 쓴 프롬프트"도 잘 동작한다. 반면 파이프라인은 직접 설계하고 구축해야 한다. 남이 만들어주지 않는다.

---

## 시작하는 방법

파이프라인을 처음 만든다면, 작게 시작하는 게 좋다.

**1. 반복 작업 찾기** — 매일 또는 매주 비슷한 패턴으로 하는 AI 작업이 있는가? 그게 파이프라인 후보다.

**2. 자동화 지점 정하기** — 전체를 자동화하지 않아도 된다. 입력만 자동화해도 된다. 출력만 자동화해도 된다.

**3. 도구 선택** — 코딩할 줄 알면 GitHub Actions나 스크립트. 모르면 n8n, Make, Zapier.

**4. 에러 대비** — AI가 이상한 결과를 낼 수 있다. 확인 단계를 넣거나, 문제가 생기면 알림을 받도록 한다.

**5. 점진적 확장** — 작은 파이프라인이 돌아가면 다음 작업을 연결한다. 블로그 글 생성 → 자동 커밋 → 자동 배포.

---

## 파이프라인이 프롬프트를 이긴다

프롬프트 엔지니어링이 필요 없다는 말이 아니다. 프롬프트는 파이프라인의 핵심 부품이다. 좋은 프롬프트가 들어간 파이프라인은 더 좋은 결과를 낸다.

하지만 프롬프트만 갈고 닦으면서 매번 수동으로 돌리는 건 삽질이다. 그 시간에 파이프라인 하나 더 만드는 게 낫다.

AI 시대의 경쟁력은 "한 번 잘하는 것"이 아니라 **"잘 굴러가게 만드는 것"**이다. 프롬프트를 10번 다듬을 시간에, 그 프롬프트가 100번 자동으로 돌아가는 구조를 만들어라.

잠자는 동안에도 일하는 사람이 이긴다.
