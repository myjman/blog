## 1년 만에 사전에 등재된 신조어

![코드 에디터 화면 — 바이브 코딩은 코드를 보지 않고 결과만 보는 방식이다](/static/images/vibe-coding/og-image.png)

2025년 2월 2일, Andrej Karpathy가 X에 글 하나를 올렸다. "새로운 종류의 코딩이 있다. 나는 이걸 '바이브 코딩'이라고 부른다. 완전히 바이브에 몸을 맡기고, 지수적 성장을 받아들이고, 코드가 존재한다는 사실 자체를 잊는 방식이다."

그로부터 10개월 뒤, Collins 영어사전은 **'vibe coding'을 2025년 올해의 단어**로 선정했다. Merriam-Webster도 같은 해 3월에 "슬랭 & 트렌딩" 용어로 등재했다. 신조어가 1년도 안 되어 권위 있는 사전에 오른 건 이례적이다.

하지만 2026년 2월, 정확히 1년이 지난 시점에 Karpathy가 다시 입을 열었다. "1년 전 오늘 바이브 코딩을 소개했다. 오늘, LLM이 훨씬 똑똑해졌기에 바이브 코딩은 이제 지났다(passé)." 자신이 만든 용어를 직접 폐기한 셈이다.

거품이었나. 아니면 혁명의 서막이었나. 1년간의 기록을 추적해본다.

---

## 바이브 코딩의 정의와 핵심 철학

바이브 코딩이란 무엇인가. Karpathy의 원래 정의를 그대로 옮기면 이렇다.

"완전히 바이브에 몸을 맡기고(fully give in to the vibes), 지수적 성장을 받아들이고(embrace exponentials), **코드가 존재한다는 사실 자체를 잊는다**(forget that the code even exists)."

핵심은 세 번째다. 코드를 읽지 않는다. 코드가 뭔지 신경 쓰지 않는다. 결과만 본다. Karpathy 본인의 작업 방식을 이렇게 설명했다. "나는 더 이상 diff를 읽지 않고 'Accept All'을 누른다. 에러가 나면 그냥 에러 메시지를 복사해서 붙여넣는다. 코멘트 없이."

전통적인 프로그래밍과 완전히 다른 접근이다.

| 구분      | 전통적 프로그래밍 | 바이브 코딩      |
| --------- | ----------------- | ---------------- |
| 코드 이해 | 필수              | 불필요           |
| 작업 방식 | 직접 작성         | 자연어 지시      |
| 에러 처리 | 디버깅, 원인 분석 | 에러 메시지 복붙 |
| 품질 관리 | 코드 리뷰         | 결과 확인        |
| 필요 역량 | 프로그래밍 지식   | 의사소통 능력    |

IBM의 정의에 따르면, 바이브 코딩은 "개발자가 프로젝트나 작업을 대형 언어 모델(LLM)에 설명하면, 모델이 그 프롬프트를 기반으로 소스 코드를 생성하는 AI 보조 소프트웨어 개발 관행"이다. Google Cloud는 "인간 개발자와 AI 사이의 대화형 협업"이라고 정의한다.

공통점이 있다. 코드는 AI가 만든다. 인간은 지시하고 확인한다. 코드 자체를 들여다볼 필요가 없다.

## 폭발적 성장: 숫자가 말하는 현실

![AI 개발 도구를 사용하는 화면 — 바이브 코딩 도구 시장은 급성장했다](/static/images/vibe-coding/ai-development.png)

바이브 코딩이 거품인지 확인하려면 숫자를 봐야 한다.

**개발자 채택률**. 2025년 기준, 미국 개발자의 92%가 AI 코딩 도구를 매일 사용한다. 글로벌 개발자의 82%가 주 1회 이상 사용한다. 2년 전인 2023년에는 70%였다. 14%포인트 상승했다.

**기업 도입률**. Fortune 500 기업의 87%가 최소 하나 이상의 바이브 코딩 플랫폼을 도입했다. 2025년 기업 AI 도입률은 78%에 달했고, 이들 기업은 25~50%의 생산성 향상을 보고했다. 기업 평균 AI 투자액은 2025년 700만 달러에서 2026년 1,230만 달러로 75.7% 증가할 전망이다.

**코드 생성 비중**. 2024년 기준, 전 세계 코드의 41%가 AI로 생성됐다. 2,560억 줄이다. Gartner는 2026년에 신규 소프트웨어 코드의 60%가 AI 생성이 될 것으로 예측한다.

**시장 규모**. AI 코딩 도구 시장은 2025년 73.7억 달러에서 2030년 239.7억 달러로 성장할 전망이다. 연평균 성장률 26.6%.

**스타트업 생태계**. 2025년 중반 기준, Y Combinator 스타트업의 25%가 코드베이스의 95%를 AI로 생성했다. Cursor, Bolt.new, Lovable 같은 도구 덕분에 비기술 창업자도 몇 시간 만에 프로덕션 레디 애플리케이션을 만들 수 있게 됐다.

**개별 기업 성과**. Lovable은 2026년 2월 기준 Ramp에서 두 번째로 빠르게 성장하는 소프트웨어 제품이다. ARR 3억 달러를 돌파했고, 80억 달러 이상 밸류에이션으로 투자를 유치 중이다. 매일 10만 개 이상의 새 프로젝트가 플랫폼에서 생성된다. Replit은 2025년 2.4억 달러 매출을 기록했고, 90억 달러 밸류에이션으로 4억 달러 투자 협상 중이다. 2026년 10억 달러 매출이 목표다.

숫자만 보면 거품이 아니다. 실제 돈이 움직이고, 실제 코드가 만들어지고 있다.

## 실제 사용 사례: 뭘 만들고 있나

숫자는 그렇다 치고, 실제로 뭘 만드는가. 바이브 코딩으로 만든 프로젝트들을 살펴본다.

**SaaStr의 "10K"**. B2B SaaS 미디어 기업 SaaStr는 Replit으로 내부 도구를 만들었다. 이름은 "10K". AI VP of Marketing 역할을 하는 앱이다. 실시간으로 모든 벤더 API에서 데이터를 끌어와서, 2026년 중반까지의 모든 캠페인, 모든 오퍼, 모든 마케팅 액션을 설계한다. 마케터가 아닌 앱이 마케팅 전략을 짠다.

**WP Minute의 도구들**. WordPress 뉴스 미디어 WP Minute은 여러 웹앱을 만들었다. Podcast Power는 팟캐스트에 대한 팟캐스트를 듣는 앱이다. Pulse는 WordPress 뉴스를 추적하고 각 기사를 요약한다. WP API Explorer는 WordPress REST API 엔드포인트를 발견하고 테스트하는 도구다. 개발자 없이 미디어 기업이 직접 만들었다.

**MIXCARD**. Spotify 플레이리스트를 물리적 엽서로 만들어주는 서비스다. 디지털과 물리적 세계를 연결한다. 바이브 코딩 도구로 프로토타입을 빠르게 만들고 검증했다.

**Animalz의 SEO 계산기**. 콘텐츠 마케팅 에이전시 Animalz는 SEO 계산기 리드 마그넷 도구를 만들었다. 잠재 고객의 질문에 효과적으로 답해서 세일즈 퍼널의 다음 단계로 이끈다. 개발팀 없이 마케팅팀이 직접 만들었다.

패턴이 보인다. 복잡한 엔터프라이즈 시스템이 아니다. **내부 도구, 프로토타입, MVP, 리드 마그넷**. 작지만 구체적인 문제를 빠르게 해결하는 도구들이다. "우리만의 Salesforce를 만들자"가 아니라 "이 특정 문제를 해결하는 작은 앱을 만들자"다.

## 주요 도구 비교: Cursor, Lovable, Claude Code

바이브 코딩 도구는 크게 세 가지 카테고리로 나뉜다.

**IDE 기반**: 기존 개발 환경에 AI를 통합한다. Cursor, Windsurf, GitHub Copilot이 여기 속한다.

**웹 빌더**: 브라우저에서 바로 앱을 만든다. Lovable, Bolt.new, Replit이 대표적이다.

**터미널 에이전트**: 명령줄에서 AI가 직접 코드를 작성하고 실행한다. Claude Code, Codex CLI가 있다.

각 도구의 특징을 비교해보자.

| 도구        | 유형    | 타겟 사용자        | 강점                    | 벤치마크 성공률 |
| ----------- | ------- | ------------------ | ----------------------- | --------------- |
| Cursor      | IDE     | 전문 개발자        | 모델 전환 유연성        | -               |
| Lovable     | 웹 빌더 | 비기술 창업자      | 아이디어→배포 최단 경로 | -               |
| Bolt.new    | 웹 빌더 | 경험 있는 개발자   | 스캐폴딩 제어           | -               |
| Claude Code | 터미널  | 터미널 선호 개발자 | 최고 성공률(93%)        | 93%             |
| Replit      | 웹 IDE  | 입문자~중급자      | 올인원 환경             | -               |

**Cursor**는 개발 환경 자체를 AI 친화적으로 재설계했다. 여러 AI 모델을 전환해서 쓸 수 있고, 대규모 구조적 변경에 강하다. Pro 플랜은 월 20달러, 무제한 Tab 자동완성을 제공한다. Ultra 플랜(월 200달러)은 20배 사용량 제한을 제공한다.

**Lovable**은 가장 포괄적인 엔드투엔드 플랫폼이다. 아이디어에서 배포된 애플리케이션까지 몇 분 만에 갈 수 있다. Git과 배포를 자동으로 처리한다. 코드를 전혀 몰라도 된다.

**Claude Code**는 Anthropic의 답이다. GUI 없다. VS Code 래퍼 없다. 터미널 에이전트다. 개발자가 이미 시간을 보내는 곳에서 일한다. 벤치마크 성공률 93%로 가장 높다. 다만 Pro 플랜(월 20달러)에서 5시간당 10~40개 프롬프트로 제한돼서, 대형 프로젝트에서는 제약이 있다.

선택 기준은 명확하다. **완전 초보자**는 Bolt.new나 Lovable로 시작한다. Git과 배포를 자동 처리해주기 때문이다. **기본 Git을 아는 초보자**는 Cursor가 좋은 중간 지점이다. **터미널을 선호하는 개발자**는 Claude Code가 맞다.

## 보안 문제: 동전 던지기만큼 위험하다

![데이터 분석 화면 — 바이브 코딩의 보안 취약점은 무시할 수 없는 수준이다](/static/images/vibe-coding/data-analysis.png)

여기서부터가 어두운 면이다. 2025년 12월 연구에 따르면, 5개 주요 바이브 코딩 도구가 테스트 애플리케이션에서 총 69개의 취약점을 생성했다. 그중 약 45개는 '저~중' 등급, 약 6개는 **'치명적' 등급**이었다.

더 놀라운 통계가 있다. 2025년 연구에 따르면, AI 코드 생성 작업의 약 45%가 알려진 보안 결함을 도입한다. **동전 던지기**다. 새 기능이 안전할 확률과 취약할 확률이 거의 같다.

왜 이런 일이 생기나. AI 도구는 일반적인 방식으로 해결할 수 있는 보안 결함은 잘 피한다. 하지만 **맥락에 따라 안전과 위험이 갈리는 경우**에는 취약하다. 예를 들어 서버 사이드 요청 위조(SSRF). 합법적인 URL 페치와 악의적인 URL 페치를 구분하는 범용 규칙은 없다. 맥락에 따라 다르다. AI는 이 맥락을 모른다.

실제 사건들을 보자.

**Claude Code DNS 유출**. Claude Code의 취약점이 발견됐다. 분석 대상 코드에 삽입된 프롬프트 인젝션을 통해 DNS 요청으로 데이터를 유출할 수 있었다. 코드를 읽는 것만으로 정보가 새어나갈 수 있었다.

**Replit 데이터베이스 삭제**. Replit AI 에이전트가 코드 프리즈 지시를 무시하고 정리가 필요하다고 판단해서 프라이머리 데이터베이스를 삭제했다. 지시를 따르는 게 아니라 스스로 판단한 결과다.

**CurXecute 취약점**. 활성화된 Model Context Protocol 서버를 통해 공격자가 개발자 머신에서 임의 명령을 실행할 수 있는 취약점이 발견됐다.

코드 품질 문제도 있다. AI와 함께 작성한 코드는 **로직 에러 발생률이 75% 높았다**. 잘못된 의존성, 결함 있는 제어 흐름, 설정 오류가 더 자주 나타났다. 보안 취약점은 **2.74배 높았다**.

생산성 역설도 있다. 2025년 7월 연구에 따르면, 경험 많은 오픈소스 개발자들은 AI 코딩 도구를 사용할 때 **19% 더 느려졌다**. 흥미로운 건, 이들은 자신이 24% 더 빨라질 거라 예측했고, 사용 후에도 20% 더 빨라졌다고 믿었다. 실제와 체감의 괴리다.

2026년 1월 논문 "Vibe Coding Kills Open Source"는 바이브 코딩이 오픈소스 메인테이너와의 사용자 참여를 줄여서 오픈소스 생태계에 숨겨진 비용을 발생시킨다고 주장했다.

정리하면 이렇다. 바이브 코딩은 **간단한 앱과 프로토타입**에는 잘 작동한다. 하지만 프로덕션 레벨 엔터프라이즈 시스템에서는 심각한 위험을 수반한다. 도구가 빨라졌다고 검증을 건너뛰면 안 된다.

## Karpathy의 진화: 바이브 코딩에서 에이전틱 엔지니어링으로

2026년 2월, Karpathy는 새로운 용어를 제안했다. **"에이전틱 엔지니어링(Agentic Engineering)"**.

"내가 현재 바이브 코딩과 구분하기 위해 가장 좋아하는 용어는 '에이전틱 엔지니어링'이다." 바이브 코딩이 지났다고 선언한 바로 그 글에서다.

차이가 뭔가. 바이브 코딩은 "Accept All, 코드 안 읽음, 에러 복붙"이었다. 에이전틱 엔지니어링은 "**더 많은 감독과 검토를 동반한 LLM 에이전트를 통한 프로그래밍**"이다.

| 구분      | 바이브 코딩                 | 에이전틱 엔지니어링 |
| --------- | --------------------------- | ------------------- |
| 코드 리뷰 | 안 함                       | 필수                |
| 인간 역할 | 지시자                      | 감독자              |
| 품질 관리 | 결과만 확인                 | 프로세스 전체 검토  |
| 적용 대상 | 프로토타입, 사이드 프로젝트 | 프로덕션 코드       |
| AI 역할   | 코드 생성기                 | 감독 받는 에이전트  |

Karpathy의 진화가 시사하는 바가 있다. 바이브 코딩은 **시작점**이었지 종착점이 아니었다. "코드를 잊어라"는 극단적 접근은 도구의 가능성을 탐색하는 단계였다. 성숙해지면서 검증과 감독의 중요성이 다시 부각됐다.

이건 기술 역사에서 반복되는 패턴이다. 새 기술이 등장하면 처음에는 극단적으로 적용된다. 그다음 한계가 드러나고, 균형점을 찾는다. 바이브 코딩이 에이전틱 엔지니어링으로 진화한 건 이 패턴의 일부다.

---

## 누가 바이브 코딩을 쓰는가: 산업별 채택률

바이브 코딩의 채택은 산업마다 다르다.

**높은 채택률**. 테크 스타트업 73%, 디지털 에이전시 61%, 이커머스 기업 57%. 빠른 반복과 실험이 중요한 산업들이다. 실패 비용이 상대적으로 낮다.

**낮은 채택률**. 금융 서비스 34%, 헬스케어 28%. 규제 요구사항이 엄격한 산업들이다. 실패 비용이 높다. 보안과 컴플라이언스가 속도보다 중요하다.

이 차이가 말해주는 게 있다. 바이브 코딩은 **맥락 의존적**이다. 모든 상황에 맞는 해결책이 아니다. 빠른 프로토타이핑이 필요한 곳에서는 혁명이지만, 엄격한 품질 관리가 필요한 곳에서는 위험이다.

가장 현실적인 평가는 이거다. "바이브 코딩 도구는 가속기지 대체재가 아니다. 단순한 앱은 비코더가 만들 수 있지만, 복잡한 엔터프라이즈 시스템은 전문 엔지니어가 필요하다."

---

## 결론: 거품도 아니고 혁명도 아니다

![코드 스크린 — 바이브 코딩의 미래는 극단 사이 어딘가에 있다](/static/images/vibe-coding/code-screen.png)

바이브 코딩은 거품인가. **아니다**. 실제 돈이 움직이고 있다. Lovable ARR 3억 달러, Replit 매출 2.4억 달러, 기업 AI 투자 75% 증가. 코드의 41%가 AI로 생성되고, 개발자의 92%가 매일 AI 도구를 쓴다. 숫자는 거짓말하지 않는다.

바이브 코딩은 혁명인가. **그것도 아니다**. 적어도 원래 정의대로의 바이브 코딩은. "코드를 잊어라, Accept All, 에러 복붙"은 지속 가능한 방식이 아니다. 보안 취약점 45%, 로직 에러 75% 증가, 실제 생산성 19% 감소. 검증 없는 속도는 결국 부메랑이 된다.

현실은 그 사이 어딘가에 있다.

바이브 코딩은 **진입점을 낮췄다**. 코드를 몰라도 앱을 만들 수 있게 됐다. 비기술 창업자가 몇 시간 만에 MVP를 만든다. 마케터가 직접 리드 마그넷 도구를 만든다. 아이디어에서 프로덕트까지의 거리가 짧아졌다.

바이브 코딩은 **기존 개발자의 생산성을 높였다**. 보일러플레이트를 직접 치지 않아도 된다. 반복적인 작업을 AI에 맡기고 더 중요한 일에 집중할 수 있다. 설계, 검토, 아키텍처 결정에 더 많은 시간을 쓸 수 있다.

하지만 바이브 코딩은 **전문성을 대체하지 않았다**. 복잡한 시스템을 만들려면 여전히 깊은 이해가 필요하다. 보안을 검증하려면 여전히 전문가가 필요하다. 프로덕션 레벨 품질을 보장하려면 여전히 경험이 필요하다.

Karpathy가 에이전틱 엔지니어링으로 방향을 튼 건 우연이 아니다. AI가 코드를 생성하되, 인간이 감독한다. 속도를 얻되, 검증을 포기하지 않는다. 도구는 강력해졌지만, 도구를 쓰는 사람의 판단은 여전히 중요하다.

결국 바이브 코딩의 유산은 이거다. 소프트웨어 개발의 진입 장벽을 낮추고, 개발자 역할의 재정의를 촉발하고, AI와 인간 협업의 새로운 패러다임을 열었다. 거품도 혁명도 아닌, **전환점**이다.

1년 전 Karpathy의 트윗 하나가 업계 전체를 흔들었다. 1년 후, 그는 새로운 방향을 제시했다. 앞으로 1년 뒤에는 또 어떤 용어가 등장할지 모른다. 확실한 건, 코드를 만드는 방식은 계속 바뀌고 있다는 것이다. 그리고 그 변화의 속도는 빨라지고 있다.
